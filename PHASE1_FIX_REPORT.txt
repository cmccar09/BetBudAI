ISSUE IDENTIFIED
----------------
Phase 1 code deployed to Lambda but import path issues prevented signals from loading.

Error message:
  "[PHASE1] Signals not available: attempted relative import beyond top-level package"


ROOT CAUSE
----------
File: backend/core/scoring/__init__.py (lines 88-104)

Problem:
  - scoring/__init__.py used relative import: "from ..signals import"
  - Lambda deployment creates flat structure: sf_analysis.py, scoring/, signals/
  - In flat structure, there's no parent package for ".." to reference
  - Import failed, Phase 1 signals fell back to stub functions (return 0)


DEPLOYMENT STRUCTURE
-------------------
Lambda package layout (flat):
  sf_analysis.py          <- Handler
  scoring/__init__.py     <- Tried "from ..signals" (FAILED)
  scoring/*.py
  signals/__init__.py     <- Phase 1 signals
  signals/*.py
  utils/*.py

Local project structure (nested):
  backend/
    lambda/sf_analysis.py
    core/
      scoring/__init__.py   <- Uses "from ..signals" (works locally)
      signals/__init__.py


FIX APPLIED
-----------
File: backend/core/scoring/__init__.py

Changed:
  try:
      from ..signals import (...)  # Works locally
      PHASE1_SIGNALS_AVAILABLE = True
  except ImportError as e:
      PHASE1_SIGNALS_AVAILABLE = False
      # Fallback stubs

To:
  try:
      from ..signals import (...)  # Works locally
      PHASE1_SIGNALS_AVAILABLE = True
  except ImportError:
      try:
          from signals import (...)  # Lambda flat structure
          PHASE1_SIGNALS_AVAILABLE = True
      except ImportError as e:
          PHASE1_SIGNALS_AVAILABLE = False
          # Fallback stubs
